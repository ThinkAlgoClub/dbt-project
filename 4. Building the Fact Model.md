## 4. Building the Fact Model – `fct_enrollments`

### What is a Fact Table?

A **fact** table records **measurable events** or **periods** over time.  
- `fct_enrollments` = one row per **enrollment period** (a continuous stretch at one school).  
- It answers questions like: "How many students started in September?" or "What's the average duration of enrollment?"  
- Unlike `dim_students` (which shows *current* state), the fact table shows *all* historical periods.

### The Grain (Most Important Decision)

**Grain = What does one row represent?**

For `fct_enrollments`, the correct grain is:  
**One row per student per continuous enrollment at a single school.**

Let's test this grain with your data:

| Student | Scenario | Number of rows in fact table |
|---------|----------|------------------------------|
| STU001 (Maria) | One school all year | 1 row |
| STU003 (Aisha) | Transfer: Quest → Zenith | 2 rows (Quest period, Zenith period) |
| STU008 (Liam) | Overlapping! (Rise until Oct 15, Frontier from Oct 10) | This is a data problem – we'll need to fix or flag |

### Handling Overlapping Enrollments (The Liam Problem)

Look at `STU008, Liam Martinez`:

| school_name | entry_date | exit_date | status |
|-------------|------------|-----------|--------|
| IDEA Rise Academy | 2025-08-18 | 2025-10-15 | Withdrawn |
| IDEA Frontier Academy | 2025-10-10 | (blank) | Active |

**The problem:** Oct 10 to Oct 15 is overlap (5 days). A student cannot physically be at two schools simultaneously.

**Why does this happen in real life?**
- Data entry error: The new school recorded entry before the old school processed withdrawal.
- Late withdrawal entry: Liam actually left Rise on Oct 9, but staff entered Oct 15.

**Analytics Engineer's job:** Decide how to resolve this for reporting.

**Three common strategies:**
1. **Adjust dates** (preferred): Shift the earlier exit_date to the day before the next entry (Oct 9). This preserves non-overlap.
2. **Flag and keep both** (honest but messy): Keep overlap, add a column `is_overlapping`. Report builders must filter.
3. **Delete the shorter period** (not recommended): Keep only the longer enrollment.

**For your interview, choose Strategy 1 (adjust dates)** – it's the most common in production dbt models.

### The SQL for Fact Model (With Overlap Resolution)

```sql
-- File: models/facts/fct_enrollments.sql
-- Materialization: incremental (append new enrollments, update changed ones)

WITH stg_enrollments AS (
    SELECT * FROM {{ ref('stg_sis_enrollments') }}
),

-- Step 1: For each student, look at next enrollment start date
with_next_start AS (
    SELECT 
        *,
        LEAD(entry_date_clean) OVER (
            PARTITION BY student_id 
            ORDER BY entry_date_clean
        ) AS next_entry_date
    FROM stg_enrollments
),

-- Step 2: Adjust exit_date if it overlaps with next enrollment
adjusted_dates AS (
    SELECT 
        student_id,
        first_name,
        last_name,
        school_id,
        school_name,
        grade_level_int,
        entry_date_clean AS enrollment_start_date,
        CASE 
            -- If exit_date is NULL, student is still at this school (current)
            WHEN exit_date_clean IS NULL THEN NULL
            -- If exit_date is after next enrollment start, cap it to day before
            WHEN next_entry_date IS NOT NULL 
                 AND exit_date_clean >= next_entry_date 
            THEN next_entry_date - INTERVAL '1 day'
            -- Otherwise keep original exit_date
            ELSE exit_date_clean
        END AS enrollment_end_date,
        entry_code,
        exit_code,
        status,
        -- Flag to indicate we adjusted (for auditing)
        CASE 
            WHEN exit_date_clean IS NOT NULL 
                 AND next_entry_date IS NOT NULL 
                 AND exit_date_clean >= next_entry_date 
            THEN TRUE 
            ELSE FALSE 
        END AS was_end_date_adjusted
    FROM with_next_start
)

SELECT 
    {{ dbt_utils.generate_surrogate_key([
        'student_id', 
        'school_id', 
        'enrollment_start_date'
    ]) }} AS enrollment_fact_id,
    student_id,
    school_id,
    school_name,
    grade_level_int AS grade_at_enrollment,
    enrollment_start_date,
    enrollment_end_date,
    -- Derived: days enrolled (NULL for open enrollments)
    DATEDIFF('day', enrollment_start_date, 
        COALESCE(enrollment_end_date, CURRENT_DATE())
    ) AS days_enrolled_to_date,
    entry_code,
    exit_code,
    status,
    was_end_date_adjusted,
    -- Boolean: is this enrollment currently active?
    CASE 
        WHEN enrollment_end_date IS NULL THEN TRUE 
        ELSE FALSE 
    END AS is_open_enrollment,
    CURRENT_TIMESTAMP() AS fact_updated_at
FROM adjusted_dates
WHERE enrollment_start_date IS NOT NULL
```

### What Happens to Liam After Adjustment?

**Before adjustment:**
| school | start | end | overlap |
|--------|-------|-----|---------|
| Rise | 2025-08-18 | 2025-10-15 | (end after next start) |
| Frontier | 2025-10-10 | NULL | |

**After adjustment:**
| school | start | end | adjusted? |
|--------|-------|-----|-----------|
| Rise | 2025-08-18 | 2025-10-09 | TRUE (end moved from Oct 15 to Oct 9) |
| Frontier | 2025-10-10 | NULL | FALSE |

Now the enrollment periods are contiguous (no gap, no overlap). The student was at Rise until Oct 9, then started Frontier on Oct 10.

### Materialization: Incremental Table

Why **incremental**?
- You have 87,000 students, each with maybe 1-3 enrollment periods per year. Full rebuild daily is fine, but incremental is more efficient.
- New enrollments come in daily (new students, transfers). You only need to process new or changed rows.

**dbt incremental config:**
```sql
{{
    config(
        materialized='incremental',
        unique_key='enrollment_fact_id',
        on_schema_change='append_new_columns'
    )
}}

-- In incremental runs, only process rows where entry_date > last run max date
{% if is_incremental() %}
    AND entry_date_clean > (SELECT MAX(enrollment_start_date) FROM {{ this }})
{% endif %}
```

### How to Explain in the Interview

> *"For `fct_enrollments`, I chose the grain: one row per student per continuous enrollment at a single school. I materialized it as **incremental** because enrollment data grows daily and full refreshes would be wasteful. To handle the real-world problem of overlapping transfers (example: student STU008), I implemented an overlap resolution: when a student's exit_date is after their next enrollment start date, I cap the exit_date to the day before the next start. This ensures no student appears at two schools simultaneously. I also added a `was_end_date_adjusted` flag so downstream users know when data was cleaned."*
