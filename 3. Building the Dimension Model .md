## 3. Building the Dimension Model – `dim_students`

### What is a Dimension Table?

A **dimension** table describes *who*, *what*, or *where*.  
- `dim_students` = one row per **unique student**.  
- It contains **static or slowly changing attributes** (name, current school, current grade, current status).  
- It does **NOT** contain repeated event history (that's the fact table).

In your CSV, students like `STU003` (Aisha) appear in **two rows**. Your dimension table must collapse those two rows into **one row** for Aisha, showing her **current** school and status.

### The Hard Part: "Current" is Tricky

Look at Aisha's data after staging:

| student_id | school_name | entry_date_clean | exit_date_clean | status |
|------------|-------------|------------------|-----------------|--------|
| STU003 | IDEA Quest College Prep | 2025-08-18 | 2025-11-03 | Withdrawn |
| STU003 | IDEA Zenith Academy | 2025-11-04 | NULL | Active |

**Question:** What is Aisha's "current" school today (assuming today is after Nov 4)?  
Answer: Zenith Academy, because her most recent enrollment has no exit date.

But what if a student withdrew and never re-enrolled? Then the most recent enrollment has an exit date, and status is "Withdrawn" or "Transferred".

**Rule for current enrollment:** Choose the row with the **latest start date** (entry_date_clean) among all rows for that student, but also ensure that if there is a row with no exit date, that is definitely current. In reality, you sort by entry date descending, pick the first row, and check if exit_date is NULL or after today.

### The SQL: Using Window Functions (New Concept)

A **window function** lets you look at other rows without collapsing them. Here, we will use `ROW_NUMBER()` to number each student's enrollments from newest to oldest, then pick #1.

```sql
-- File: models/dimensions/dim_students.sql
-- Materialization: table (refreshed daily or incremental)

WITH stg_enrollments AS (
    SELECT * FROM {{ ref('stg_sis_enrollments') }}
),

ranked_enrollments AS (
    SELECT 
        *,
        ROW_NUMBER() OVER (
            PARTITION BY student_id 
            ORDER BY entry_date_clean DESC
        ) AS enrollment_rank
    FROM stg_enrollments
),

current_enrollment AS (
    SELECT 
        student_id,
        first_name,
        last_name,
        school_id AS current_school_id,
        school_name AS current_school_name,
        state_code AS current_state,
        grade_level_int AS current_grade,
        entry_date_clean AS current_enrollment_start_date,
        exit_date_clean AS current_enrollment_end_date,
        status AS current_status,
        -- Business rule: a student is "currently enrolled" if:
        -- 1. They have no exit_date (still active), OR
        -- 2. Their exit_date is in the future (but data is usually current date).
        -- We'll simplify: active if exit_date IS NULL AND status = 'Active'
        CASE 
            WHEN exit_date_clean IS NULL AND status = 'Active' THEN TRUE
            ELSE FALSE
        END AS is_currently_enrolled
    FROM ranked_enrollments
    WHERE enrollment_rank = 1
)

SELECT 
    {{ dbt_utils.generate_surrogate_key(['student_id']) }} AS student_dim_id,  -- surrogate key
    student_id AS natural_student_id,
    first_name,
    last_name,
    current_school_id,
    current_school_name,
    current_state,
    current_grade,
    current_enrollment_start_date,
    current_enrollment_end_date,
    current_status,
    is_currently_enrolled,
    CURRENT_TIMESTAMP() AS dim_updated_at
FROM current_enrollment
```

### Why This Works for Transfers

For Aisha:
- Two rows. Partition by `student_id`. Order by `entry_date_clean DESC`:
  - Row 1: entry 2025-11-04 (Zenith) → rank 1
  - Row 2: entry 2025-08-18 (Quest) → rank 2
- We pick rank 1. That gives her current school = Zenith, current_status = Active.

For a student who transferred twice? Same logic. Always the latest start.

### What About Students Who Leave and Never Return?

Look at `STU020, Alexander Allen`:
- One row only. Exit date 2025-11-22, status `Withdrawn`.
- Rank = 1. Exit date is NOT NULL. `is_currently_enrolled` becomes FALSE.
- That's correct: he is not currently enrolled anywhere.

### Materialization: Table, not View

`dim_students` should be a **table** (refreshed daily or incrementally). Why?
- Downstream models (facts, reports) will join to it repeatedly. A table is fast.
- A view would recalculate the "current enrollment" logic every time, which is expensive.
- In dbt, you can schedule it to rebuild every night when new enrollment data arrives.

**Incremental?** Yes, if you have millions of students. For 87k, a full refresh table is fine.

### How to Explain in the Interview

> *"For `dim_students`, I chose a **table** materialization because this is a downstream lookup that many reports will join. The grain is one row per student. To handle transfers, I used a window function `ROW_NUMBER()` partitioned by student_id and ordered by entry_date descending, taking the most recent enrollment as the current record. I also added a boolean flag `is_currently_enrolled` based on exit_date being NULL and status 'Active'. This allows stakeholders to easily count currently enrolled students."*
