## 5. Testing Your Models – dbt YAML and Custom Tests

### Why Testing Matters (The "Trust" Problem)

Imagine you send the superintendent a report saying "87,000 students are enrolled." But your data has:
- Duplicate student IDs (counts would be wrong)
- NULL school names (can't tell which campus)
- Overlapping enrollments (same student counted twice)
- Invalid state codes like "Texs" (breaks filtering)

The superintendent loses trust. Your job as an Analytics Engineer is to **automatically catch these errors** before anyone sees the report.

### Two Types of Tests in dbt

1. **Schema tests** (built-in): Check column-level rules like `unique`, `not_null`, `accepted_values`, `relationships`.
2. **Custom data tests** (you write SQL): Check business rules like "no overlapping enrollment periods".

### Part 2 of Your Interview: Write the YAML Test Definitions

You will create a `schema.yml` file (or multiple files) that dbt reads. Let's write tests for each model.

---

### Tests for `stg_sis_enrollments` (Staging)

Staging should be clean but not yet business-rules enforced. Focus on **data integrity**.

```yaml
# File: models/staging/schema.yml

version: 2

models:
  - name: stg_sis_enrollments
    description: "Cleaned raw enrollment data, one row per raw record"
    columns:
      - name: enrollment_id
        description: "Surrogate key for staging"
        tests:
          - unique
          - not_null
      
      - name: student_id
        description: "Natural key from source"
        tests:
          - not_null
      
      - name: school_id
        tests:
          - not_null
      
      - name: state_code
        description: "Two-letter state code"
        tests:
          - accepted_values:
              values: ['TX', 'OH', 'FL']
              # If any other value appears, fail the test
      
      - name: grade_level_int
        description: "Grade level as integer, K=0, NULL allowed for missing"
        tests:
          - dbt_utils.accepted_range:
              min_value: 0
              max_value: 12
              # Kindergarten = 0, grade levels 1-12. N/A becomes NULL so not tested.
      
      - name: entry_date_clean
        tests:
          - not_null
          - dbt_utils.not_null_proportion:
              at_least: 0.99  # Allow very few NULLs but warn
      
      - name: exit_date_clean
        tests:
          - dbt_utils.not_null_proportion:
              at_least: 0.70  # Expect ~30% NULL because active students have no exit date
      
      - name: status
        tests:
          - accepted_values:
              values: ['Active', 'Withdrawn', 'Transferred']
```

**Why these tests?**
- `unique` + `not_null` on `enrollment_id`: Ensures every raw row has a unique identifier.
- `accepted_values` on `state_code`: Catches "Texas" or "Tex" – forces standardization.
- `accepted_range` on `grade_level_int`: Ensures no grade 99 or negative.
- `not_null_proportion` on `exit_date`: Recognizes that NULL is expected for active students, but if too many NULLs or too few, something's wrong.

### Tests for `dim_students` (Dimension)

Dimension should have **one row per student** and **current attributes must be valid**.

```yaml
# File: models/dimensions/schema.yml

models:
  - name: dim_students
    description: "One row per student, current enrollment snapshot"
    columns:
      - name: student_dim_id
        tests:
          - unique
          - not_null
      
      - name: natural_student_id
        tests:
          - unique          # Critical: each student appears only once
          - not_null
          - relationships:
              to: ref('stg_sis_enrollments')
              field: student_id
              # Ensures every dim student exists in staging (referential integrity)
      
      - name: current_school_id
        tests:
          - not_null
          - relationships:
              to: ref('stg_sis_enrollments')
              field: school_id
      
      - name: is_currently_enrolled
        tests:
          - not_null
          - dbt_utils.not_null_proportion:
              at_least: 0.50  # At least half are currently enrolled (reasonable)
    
    # Custom test: ensure no student has two rows (already covered by unique on natural_student_id)
```

**Why the `relationships` test?**
- It checks that every `student_id` in `dim_students` actually exists in `stg_sis_enrollments`. If not, you have a phantom student.

### Tests for `fct_enrollments` (Fact)

Fact needs the most rigorous tests because it's used for counting.

```yaml
# File: models/facts/schema.yml

models:
  - name: fct_enrollments
    description: "One row per continuous enrollment period, with overlap resolution"
    columns:
      - name: enrollment_fact_id
        tests:
          - unique
          - not_null
      
      - name: student_id
        tests:
          - not_null
          - relationships:
              to: ref('dim_students')
              field: natural_student_id
      
      - name: school_id
        tests:
          - not_null
      
      - name: enrollment_start_date
        tests:
          - not_null
      
      - name: enrollment_end_date
        tests:
          - dbt_utils.not_null_proportion:
              at_least: 0.70  # Some are NULL (open enrollments)
      
      - name: is_open_enrollment
        tests:
          - not_null
          - dbt_utils.not_null_proportion:
              at_least: 0.50
```

### The Most Important Test: Custom Data Test for Overlapping Enrollments

The built-in tests cannot check a complex business rule like:  
**"No student should have overlapping enrollment periods at different schools."**

You write a **custom test** as a SQL query that returns rows that violate the rule. If it returns any rows, the test fails.

**File:** `tests/no_overlapping_enrollments.sql`

```sql
-- Custom test: Find any student with overlapping enrollment periods
-- After overlap resolution, this should return zero rows.
-- If it returns rows, our resolution logic failed or raw data is impossible.

SELECT 
    a.student_id,
    a.school_id AS school_a_id,
    a.enrollment_start_date AS a_start,
    a.enrollment_end_date AS a_end,
    b.school_id AS school_b_id,
    b.enrollment_start_date AS b_start,
    b.enrollment_end_date AS b_end
FROM {{ ref('fct_enrollments') }} a
JOIN {{ ref('fct_enrollments') }} b 
    ON a.student_id = b.student_id
    AND a.enrollment_fact_id < b.enrollment_fact_id  -- avoid duplicate pairs
WHERE 
    -- Overlap condition: a's period overlaps with b's period
    a.enrollment_start_date <= COALESCE(b.enrollment_end_date, '9999-12-31')
    AND b.enrollment_start_date <= COALESCE(a.enrollment_end_date, '9999-12-31')
    -- Exclude same school (student could re-enroll later at same school without overlap)
    AND a.school_id != b.school_id
```

**What this test does:**  
It finds any two enrollment periods for the same student at different schools where the date ranges overlap. After your overlap resolution (from Lesson 4), this should never happen. If it does, your raw data has a problem that your adjustment didn't fix.

**In the interview, say:**  
> *"I chose a custom test for overlapping enrollments because it's a critical business rule: a student cannot be enrolled at two campuses simultaneously. This test would run after my fact model builds. If it fails, I would investigate the raw data for data entry errors and potentially notify the school operations team."*

---

### How to Present the Test Suite (Part 2 of Interview)

You will say:

> *"I've defined three types of tests. First, **schema tests** on staging: uniqueness, not-null, accepted values for state codes, and a range for grade levels. Second, **relationships tests** on dimensions to ensure referential integrity between `dim_students` and staging. Third, a **custom data test** that checks for overlapping enrollments across different schools – this is the most important business rule for enrollment. Each test has a clear failure meaning: for example, if `accepted_values` on `state_code` fails, it means our cleaning logic missed a non-standard state name like 'Texas'. I would set up dbt to run these tests every time the models are built, and alert the team via email or Slack on failures."*
