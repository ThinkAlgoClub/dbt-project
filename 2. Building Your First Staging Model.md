## 2. Building Your First Staging Model – `stg_sis_enrollments`

### The Goal of a Staging Model

A staging model is the **first stop** after raw data. Its job is simple:

1.  **Rename** columns to be clear and consistent (e.g., `student_id` stays, but raw column names might be cryptic – here they are okay).
2.  **Cast** columns to the correct data types (dates as dates, numbers as numbers, text as text).
3.  **Fix** obvious data quality issues that you can fix without business rules.
4.  **Add** a primary key if needed (for a staging table, often a surrogate key like `enrollment_id`).
5.  **Do NOT** join or aggregate yet. Keep it one-to-one with the raw CSV.

**Materialization choice for staging: `VIEW`** (or sometimes `EPHEMERAL` in dbt). Why? Because staging is just a light cleaning layer. It doesn't store data; it just shows a cleaned version of the raw table. This saves storage and compute. Only later models (dimensions/facts) become tables.

### The Raw Problems We Saw (and How to Fix in SQL)

Let's revisit the issues from your CSV:

| Problem | Example | Fix in Staging |
| :--- | :--- | :--- |
| **Date format inconsistency** | `entry_date` has `08-18-2025` (MM-DD-YYYY) and `2025-08-18` (YYYY-MM-DD) | Parse both into a true `DATE` type. |
| **State name vs. code** | `Texas` instead of `TX` | Standardize to two-letter code. |
| **Grade level 'N/A'** | `grade_level` = `N/A` for student `STU005` | Convert to `NULL` (unknown is better than fake value). |
| **Blank exit_date means active** | `exit_date` is empty string or NULL | Leave as NULL. In analytics, NULL means "still enrolled". |
| **Duplicate rows?** | No exact duplicates, but same student multiple rows (that's fine – staging keeps all) | No fix needed. Staging preserves history. |

### The dbt Staging Model SQL

Here is the SQL you would write in dbt. I'll add comments explaining **why** each line exists. This is exactly what you would show in your interview for Part 1.

```sql
-- File: models/staging/stg_sis_enrollments.sql

WITH raw AS (
    -- In dbt, you reference the raw CSV table (loaded via Fivetran)
    -- For the interview, assume 'raw_sis_enrollments' is the CSV data.
    SELECT * FROM {{ ref('raw_sis_enrollments') }}
)

SELECT
    -- Create a unique surrogate key for each row in staging.
    -- This helps with testing and debugging.
    {{ dbt_utils.generate_surrogate_key(['student_id', 'school_id', 'entry_date']) }} AS enrollment_id,
    
    -- Keep natural keys as they are
    student_id,
    first_name,
    last_name,
    school_id,
    school_name,
    
    -- STANDARDIZE STATE: 'Texas' -> 'TX', others already 'TX','OH','FL'
    CASE 
        WHEN state = 'Texas' THEN 'TX'
        WHEN state = 'Texas ' THEN 'TX'  -- just in case of trailing spaces
        ELSE state
    END AS state_code,
    
    -- HANDLE GRADE_LEVEL: convert 'N/A' to NULL, cast remainder as integer
    CASE 
        WHEN grade_level = 'N/A' THEN NULL
        ELSE grade_level::INT
    END AS grade_level_int,
    
    -- FIX DATES: try YYYY-MM-DD first, then MM-DD-YYYY
    -- Snowflake's TRY_TO_DATE attempts multiple formats if you specify.
    -- For portability, we show explicit logic.
    CASE 
        WHEN entry_date LIKE '____-__-__' THEN entry_date::DATE
        WHEN entry_date LIKE '__-__-____' THEN 
            TO_DATE(entry_date, 'MM-DD-YYYY')
        ELSE NULL
    END AS entry_date_clean,
    
    -- Similar for exit_date (most are blank/NULL)
    CASE 
        WHEN exit_date IS NULL OR exit_date = '' THEN NULL
        WHEN exit_date LIKE '____-__-__' THEN exit_date::DATE
        WHEN exit_date LIKE '__-__-____' THEN TO_DATE(exit_date, 'MM-DD-YYYY')
        ELSE NULL
    END AS exit_date_clean,
    
    -- Keep codes as they are (but trim whitespace)
    TRIM(entry_code) AS entry_code,
    TRIM(exit_code) AS exit_code,
    TRIM(status) AS status,
    
    -- Add a loaded timestamp for debugging (in production, you'd use current_timestamp)
    CURRENT_TIMESTAMP() AS staged_at

FROM raw

-- Optional: filter out rows that are completely invalid (e.g., no student_id)
WHERE student_id IS NOT NULL
```

### What About Overlaps or Transfers? (Not in Staging)

Notice: I did **not** try to handle Aisha's transfer here. Staging keeps both rows. Why? Because business logic (like "combine overlapping enrollments") belongs in **downstream models** (facts/dimensions), not staging. Staging must remain a simple, predictable, one-to-one mirror of the raw data.

### How to Explain Your Decisions in the Interview

When you present, say something like:

> *"I chose to materialize `stg_sis_enrollments` as a **view** because it's a lightweight transformation that doesn't need to be stored. This follows dbt best practices: staging views, intermediate tables, and final facts/dimensions as tables or incrementals. I fixed three data quality issues: inconsistent date formats, the 'Texas' vs 'TX' mismatch, and the 'N/A' grade level. I did NOT try to de-duplicate or handle transfers at this stage because staging should be a clean but exact representation of raw data."*
